## 1. 进程调度算法深度对比

### 1.1 调度算法全景

```
调度算法分类:

  +-------------------+-------------------+
  |     非抢占式       |      抢占式        |
  +-------------------+-------------------+
  | FCFS              | RR (时间片轮转)    |
  | SJF (非抢占)      | SRTF (抢占SJF)    |
  | HRRN (高响应比)   | 优先级调度(抢占)   |
  +-------------------+-------------------+

  批处理系统关注: 吞吐量、周转时间
  交互式系统关注: 响应时间、公平性
  实时系统关注:   截止时间、可预测性
```

### 1.2 FCFS (先来先服务)

```
FCFS (First Come First Served):

  特点: 按到达顺序排队，非抢占
  实现: FIFO队列

  示例 (到达顺序: P1, P2, P3):
    进程   到达时间   执行时间
    P1       0         24
    P2       0          3
    P3       0          3

  甘特图:
    |---- P1(24) ----|P2(3)|P3(3)|
    0               24   27   30

  等待时间: P1=0, P2=24, P3=27
  平均等待时间: (0+24+27)/3 = 17
  平均周转时间: (24+27+30)/3 = 27

  护航效应 (Convoy Effect):
    短进程被迫等待长进程
    如: 多个I/O密集进程等待一个CPU密集进程

  优点: 简单公平，无饥饿
  缺点: 平均等待时间长，护航效应
```

### 1.3 SJF / SRTF (最短作业优先)

```
SJF (Shortest Job First) - 非抢占版:
  选择执行时间最短的进程

  示例:
    进程   到达时间   执行时间
    P1       0          7
    P2       2          4
    P3       4          1
    P4       5          4

  甘特图:
    |P1(7)|P3(1)|P2(4)|P4(4)|
    0     7    8    12    16

  P1先执行(0时刻只有P1), P3最短先执行

SRTF (Shortest Remaining Time First) - 抢占版:
  新进程到达时，若剩余时间更短则抢占

  示例 (同上):
    |P1(2)|P3(1)|P2(4)|P4(4)|P1(5)|
    0    2    3     7    11    16

  t=0: P1开始
  t=2: P2到达(剩余4 < P1剩余5) -> 抢占P1
  t=4: P3到达(剩余1 < P2剩余2) -> 抢占P2
  t=5: P4到达(剩余4 > P3剩余0) -> 不抢占
  t=8: P3完成, P2剩余2, P4剩余4 -> P2继续

  SJF是最优的(平均等待时间最小)
  但需要预知执行时间(通常不可能)
  实践中用指数平均估计:
    tau_{n+1} = alpha * t_n + (1 - alpha) * tau_n
    alpha通常取0.5
```

### 1.4 RR (时间片轮转)

```
RR (Round Robin):

  特点: 抢占式，时间片轮转，公平
  实现: 就绪队列 + 定时器中断

  示例 (时间片 q=4):
    进程   执行时间
    P1       24
    P2        3
    P3        3

  甘特图:
    |P1(4)|P2(3)|P3(3)|P1(4)|P1(4)|P1(4)|P1(4)|P1(1)|
    0     4     7    10    14    18    22    26    27

  等待时间: P1=6, P2=4, P3=7
  平均等待时间: (6+4+7)/3 = 5.67

  时间片选择的影响:
    q过大 -> 退化为FCFS
    q过小 -> 上下文切换开销大
    经验值: q应略大于典型交互的CPU突发时间
    通常: 10-100ms, 上下文切换约0.1-1ms

  时间片 vs 上下文切换开销:
    q = 10ms, 切换开销 = 0.1ms
    CPU利用率 = 10/(10+0.1) ≈ 99%

    q = 1ms, 切换开销 = 0.1ms
    CPU利用率 = 1/(1+0.1) ≈ 91%
```

### 1.5 优先级调度与多级反馈队列

```
优先级调度:

  静态优先级: 创建时确定，运行期间不变
  动态优先级: 运行期间可变

  老化 (Aging): 等待时间越长，优先级越高
    防止低优先级进程饥饿

  Linux nice值: -20(最高) ~ 19(最低)
    默认0, 超级用户可设负值

多级反馈队列 (MLFQ):

  核心思想: 无需预知进程执行时间
  通过观察行为动态调整优先级

  +------------------------------------------+
  | Queue 0 (最高优先级, q=8ms)              |
  |   [P2] -> [P5] -> ...                    |
  +------------------------------------------+
  | Queue 1 (中优先级, q=16ms)               |
  |   [P1] -> [P3] -> ...                    |
  +------------------------------------------+
  | Queue 2 (最低优先级, q=FCFS)             |
  |   [P4] -> ...                            |
  +------------------------------------------+

  规则:
    1. 新进程进入最高优先级队列
    2. 用完时间片 -> 降一级
    3. 主动放弃CPU(I/O) -> 保持或提升
    4. 高优先级队列非空时不调度低优先级
    5. 定期将所有进程提升到最高优先级(防饥饿)

  效果:
    I/O密集进程: 短CPU突发 -> 保持在高优先级 -> 响应快
    CPU密集进程: 长CPU突发 -> 逐渐降到低优先级 -> 后台运行
```

**MLFQ调度伪代码**：

```python
def mlfq_schedule(queues):
    for queue in queues:  # 从高到低
        if queue.is_empty():
            continue
        process = queue.dequeue()
        run(process, queue.time_quantum)
        if process.used_full_quantum():
            # 用完时间片，降级
            lower_queue = queues[next_level(queue)]
            lower_queue.enqueue(process)
        elif process.state == BLOCKED:
            # I/O等待，保持或提升优先级
            queues[queue.level].enqueue(process)
        break  # 只调度最高非空队列
```

## 2. 内存管理机制

### 2.1 分页机制深入

```
多级页表:

  问题: 32位地址空间, 4KB页, 4B页表项
    页表项数 = 2^32 / 2^12 = 2^20 = 1M
    页表大小 = 1M * 4B = 4MB (每个进程!)

  两级页表 (x86-32):
    虚拟地址: [Directory(10)] [Page(10)] [Offset(12)]

    Page Directory (4KB, 1024项)
      |
      v
    Page Table (4KB, 1024项)
      |
      v
    Physical Frame (4KB)

  四级页表 (x86-64, 48位虚拟地址):
    虚拟地址: [PML4(9)] [PDPT(9)] [PD(9)] [PT(9)] [Offset(12)]

    PML4 -> PDPT -> PD -> PT -> Frame
    每级512项 (2^9), 页表项8字节

  Linux五级页表 (5.8+内核, 57位虚拟地址):
    PGD -> P4D -> PUD -> PMD -> PTE -> Frame
```

### 2.2 TLB与地址翻译优化

```
TLB (Translation Lookaside Buffer):
  页表的硬件缓存，加速地址翻译

  TLB结构:
    | Tag (VPN) | PPN | V | R | W | X | D | A | ASID |

  ASID (Address Space ID):
    标识不同进程的地址空间
    避免每次上下文切换时冲刷TLB

  TLB命中流程:
    VPN -> TLB查找 -> 命中 -> 直接获得PPN
    延迟: 1-2 cycles

  TLB未命中流程:
    VPN -> TLB未命中 -> 查页表 -> 更新TLB -> 重试
    延迟: 20-100 cycles (二级TLB) / 100-200 cycles (页表遍历)

  典型TLB参数:
    L1 ITLB: 64项, 全相联
    L1 DTLB: 64项, 全相联
    L2 STLB: 1536项, 8路组相联
    覆盖范围: 1536 * 4KB = 6MB (4KB页)
              1536 * 2MB = 3GB (2MB大页)

大页 (Huge Pages):
  减少TLB未命中，提高性能

  标准页: 4KB
  大页:   2MB (x86-64 PD映射, 减少一级页表遍历)
  巨页:   1GB (x86-64 PDPT映射, 减少两级遍历)

  Linux透明大页 (THP):
    自动将连续4KB页合并为2MB大页
    对应用透明
```

### 2.3 分段机制

```
分段 (Segmentation):
  按逻辑单位划分内存，每段有段名和长度

  段表:
    | 段号 | 基址 | 长度 | 保护位 |

  地址翻译:
    虚拟地址: [段号 | 段内偏移]

    if (offset < segment[seg_num].limit) {
        phys_addr = segment[seg_num].base + offset;
    } else {
        trap(SEGMENTATION_FAULT);
    }

分页 vs 分段:

  | 特性     | 分页           | 分段            |
  |----------|---------------|-----------------|
  | 划分依据 | 物理大小(固定) | 逻辑意义(可变)  |
  | 大小     | 固定(4KB等)    | 可变            |
  | 外部碎片 | 无             | 有              |
  | 内部碎片 | 有             | 无(几乎)        |
  | 碎片类型 | 内部碎片       | 外部碎片        |
  | 可见性   | 对程序员透明   | 对程序员可见    |

段页式:
  先分段，段内再分页
  虚拟地址: [段号 | 页号 | 偏移]
  兼顾逻辑划分和物理管理
```

### 2.4 页面置换算法深入

```
FIFO页面置换:

  示例 (3帧, 访问序列: 7,0,1,2,0,3,0,4,2,3,0,3,2,1,2,0,1,7,0,1):

  帧: [7] -> [7,0] -> [7,0,1] -> [0,1,2] -> [0,1,2] -> [1,2,3]
  缺页: 15次

  Belady异常: 更多帧反而更多缺页!
    3帧: 9次缺页
    4帧: 10次缺页 (反常!)

LRU页面置换:

  基于时间栈实现:
    每次访问将页面移到栈顶
    置换栈底页面(最久未用)

  示例 (3帧, 访问序列: 1,2,3,4,1,2,5,1,2,3,4,5):

  栈状态(顶=最近):
    访问1: [1]              缺页
    访问2: [2,1]            缺页
    访问3: [3,2,1]          缺页
    访问4: [4,3,2]  置换1   缺页
    访问1: [1,4,3]  置换2   缺页
    访问2: [2,1,4]  置换3   缺页
    访问5: [5,2,1]  置换4   缺页
    访问1: [1,5,2]          命中
    访问2: [2,1,5]          命中
    访问3: [3,2,1]  置换5   缺页
    访问4: [4,3,2]  置换1   缺页
    访问5: [5,4,3]  置换2   缺页

  LRU是栈算法，不会出现Belady异常
  但精确实现代价高(每次访问更新时间戳/栈)

Clock算法 (LRU近似):

  改进型Clock算法 (考虑引用位R和修改位M):
    (R,M) = (0,0): 最佳置换候选
    (R,M) = (0,1): 未最近使用但已修改(写回代价大)
    (R,M) = (1,0): 最近使用但未修改
    (R,M) = (1,1): 最近使用且已修改(最不该置换)

  扫描策略:
    第一轮: 找(0,0) -> 置换
    第二轮: 找(0,1), 跳过时清R位 -> 置换
    第三轮: 此时所有R=0, 找(0,0) -> 置换
    第四轮: 找(0,1) -> 必然找到
```

**Clock算法实现**：

```c
struct page_frame {
    int page_id;
    int reference_bit;  // R位
    int modify_bit;     // M位
};

int clock_replace(struct page_frame frames[], int n, int *hand) {
    while (1) {
        if (frames[*hand].reference_bit == 0) {
            int victim = *hand;
            *hand = (*hand + 1) % n;
            return victim;
        }
        frames[*hand].reference_bit = 0;  // 清零给第二次机会
        *hand = (*hand + 1) % n;
    }
}
```

### 2.5 虚拟内存与工作集模型

```
工作集模型 (Working Set):

  W(t, delta) = 在时刻t的前delta个时间引用中访问的页面集合

  工作集: 进程近期活跃使用的页面集合
  delta: 工作集窗口大小

  示例:
    访问序列: ... a b c d a b c d a b c ...
    delta = 4
    W(t) = {a, b, c, d}

    访问序列: ... a a a b b b c c c ...
    delta = 3
    W(t1) = {a}, W(t2) = {a,b}, W(t3) = {b,c}

  工作集策略:
    仅当进程的工作集在内存中时才执行
    否则挂起进程(换出), 释放其所有页面

  抖动 (Thrashing):
    进程频繁缺页，大部分时间用于页面换入换出
    原因: 并发进程过多，每进程分配帧数 < 工作集大小
    解决: 减少并发进程数(挂起部分进程)

  缺页频率 (PF) 算法:
    缺页率过高 -> 增加帧数
    缺页率过低 -> 减少帧数
    自适应调整每进程的帧分配
```

## 3. 文件系统实现

### 3.1 ext4文件系统深入

```
ext4关键特性:

1. Extent替代间接块:
   ext3: 每个数据块需要4B指针
   ext4: 一个Extent描述连续块范围

   Extent结构:
   | eh_magic | eh_entries | eh_max | eh_depth | eh_generation |
   | ee_block | ee_len     | ee_start_hi | ee_start_lo        |

   一个Extent可描述: 2^15 = 32768个连续块 = 128MB
   一级Extent树可描述: 4个Extent = 512MB
   二级Extent树可描述: 几乎无限

2. 延迟分配 (Delay Allocation):
   write()时不立即分配磁盘块
   缓存在页缓存中，flush时再分配
   优点: 减少碎片，批量分配连续块

3. 多块分配 (Multiblock Allocator):
   一次分配多个连续块
   避免逐块分配导致的碎片

4. 日志校验:
   日志区域使用校验和
   防止日志损坏导致恢复错误

5. 在线碎片整理:
   e4defrag工具
   移动文件到连续块
```

### 3.2 NTFS文件系统

```
NTFS (New Technology File System):

  核心概念: 一切都是属性 (Everything is an Attribute)

  MFT (Master File Table):
    每个文件/目录一个MFT记录 (1KB)
    小文件: 数据直接存储在MFT记录中(常驻属性)
    大文件: 数据存储在簇中(非常驻属性)

  MFT特殊文件:
    $MFT       : MFT本身
    $MFTMirr   : MFT备份(前4项)
    $LogFile   : 日志文件
    $Volume    : 卷信息
    $AttrDef   : 属性定义
    $Root      : 根目录
    $Bitmap    : 簇位图

  NTFS数据结构:
    +------------------+
    | MFT Record       |
    | +--------------+ |
    | | Standard Info| |  时间戳, 属性
    | | File Name    | |  文件名(Unicode)
    | | Security Desc| |  ACL权限
    | | Data         | |  文件数据/簇映射
    | | Index Root   | |  目录索引(B+树)
    | +--------------+ |
    +------------------+

  特性:
    ADS (Alternate Data Streams): 文件可有多个数据流
    硬链接: 多个文件名指向同一MFT记录
    压缩: 透明文件压缩(16簇一组)
    EFS: 加密文件系统(基于用户证书)
```

### 3.3 APFS文件系统

```
APFS (Apple File System):

  核心设计目标: SSD优化、快照、加密

1. Copy-On-Write (COW):
   所有修改都写入新块，不覆盖旧数据
   天然支持快照: 旧块构成快照

2. 快照 (Snapshots):
   瞬间创建，零额外空间(共享未修改块)
   Time Machine的基础

3. 克隆 (Clones):
   cp文件时只复制元数据，共享数据块
   修改时才COW复制
   几乎零时间和空间开销

4. 空间共享:
   多个卷共享同一容器的空闲空间
   无需预分配分区大小

   +----------------------------------+
   | APFS Container                   |
   |  +------+ +------+ +------+     |
   |  | Vol1 | | Vol2 | | Vol3 |     |
   |  +------+ +------+ +------+     |
   |  |     Shared Free Space    |     |
   |  +------------------------------+|
   +----------------------------------+

5. 原子写入:
   保证写入要么完全成功要么完全失败
   避免断电导致文件损坏

6. 加密:
   多密钥加密: 每个文件独立密钥
   硬件加速: 利用Apple T2/M系列芯片
```

### 3.4 文件系统性能优化

```
文件系统缓存层次:

  Application Buffer
       |
  Page Cache (页缓存)         <-- 脏页回写
       |
  Buffer Cache (块缓存)       <-- 块设备I/O
       |
  Device Internal Cache       <-- 磁盘/SSD自带缓存
       |
  Storage Media

日志 (Journal) 模式:

  1. 日志模式 (data=journal):
     数据和元数据都写入日志
     最安全，最慢

  2. 有序模式 (data=ordered, 默认):
     元数据写入日志
     数据在提交日志前写入
     兼顾安全和性能

  3. 写回模式 (data=writeback):
     仅元数据写入日志
     数据可能丢失
     最快，最不安全

日志流程:
  1. 写入事务到日志区 (Journal Write)
  2. 提交日志 (Journal Commit)
  3. 写入实际位置 (Checkpoint)
  4. 释放日志空间

崩溃恢复:
  重放日志中已提交但未checkpoint的事务
```

## 4. I/O管理

### 4.1 I/O缓冲

```
缓冲的目的:
  1. 缓和CPU与I/O设备速度不匹配
  2. 减少CPU中断次数
  3. 提高并行性

单缓冲:
  CPU处理缓冲区数据时，设备可填充另一区
  +--------+    +--------+
  | Device | -> | Buffer | -> CPU处理
  +--------+    +--------+

双缓冲:
  两个缓冲区交替使用
  +--------+    +--------+
  | Device | -> | Buf A  | -> CPU处理Buf A
  +--------+    +--------+
                | Buf B  | -> CPU处理Buf B
                +--------+
  设备填Buf A时CPU处理Buf B，反之亦然

循环缓冲:
  多个缓冲区组成环形队列
  in指针: 设备填充位置
  out指针: CPU取数据位置

  +---+---+---+---+---+---+---+---+
  |   | D | D | D | D |   |   |   |
  +---+---+---+---+---+---+---+---+
        ^out             ^in

  适用于I/O速度与处理速度接近的场景
```

### 4.2 SPOOLing技术

```
SPOOLing (Simultaneous Peripheral Operations On-Line):

  核心思想: 用磁盘作为缓冲，将独占设备虚拟为共享设备

  +------------------------------------------+
  |               磁盘                        |
  |  +-------------+  +-------------+         |
  |  | 输入井       |  | 输出井       |         |
  |  | (Input Well) |  | (Output Well)|        |
  |  +-------------+  +-------------+         |
  +------------------------------------------+
       ^                    |
       |                    v
  +---------+         +----------+
  | 输入进程 |         | 输出进程  |
  | (Spi)   |         | (Spo)    |
  +---------+         +----------+
       ^                    |
       |                    v
  +---------+         +----------+
  | 输入设备 |         | 输出设备  |
  | (读卡机) |         | (打印机)  |
  +---------+         +----------+

  打印机SPOOLing示例:
    1. 用户进程调用write() -> 数据写入磁盘输出井
    2. 用户进程立即返回(不等待打印)
    3. 打印守护进程按顺序从输出井取数据送打印机
    4. 多个进程可"同时"打印(实际串行)

  优点:
    - 独占设备变为共享设备
    - 提高设备利用率
    - 用户进程不阻塞在慢速I/O上
```

### 4.3 I/O模型对比

```
五种I/O模型:

1. 阻塞I/O (Blocking I/O):
   调用read() -> 阻塞等待 -> 数据就绪 -> 拷贝数据 -> 返回
   |---- 等待数据 ----|---- 拷贝 ----|
   最简单，效率最低

2. 非阻塞I/O (Non-blocking I/O):
   调用read() -> 立即返回EAGAIN -> 轮询 -> 数据就绪 -> 拷贝 -> 返回
   CPU浪费在轮询上

3. I/O多路复用 (I/O Multiplexing):
   调用select/poll/epoll -> 阻塞等待任一fd就绪 -> read() -> 拷贝 -> 返回
   一个线程监控多个fd

4. 信号驱动I/O (Signal-driven I/O):
   注册SIGIO信号 -> 数据就绪时发信号 -> 在信号处理中read()

5. 异步I/O (Asynchronous I/O):
   调用aio_read() -> 立即返回 -> 内核完成所有操作 -> 通知应用
   |---- 应用继续执行 ----|
   真正的异步，无需应用参与

  对比:
  | 模型       | 等待方式 | 拷贝方式 | 阻塞点     |
  |-----------|---------|---------|-----------|
  | 阻塞I/O   | 阻塞    | 阻塞    | 等待+拷贝  |
  | 非阻塞I/O | 轮询    | 阻塞    | 拷贝      |
  | I/O多路复用| 阻塞   | 阻塞    | 等待+拷贝  |
  | 信号驱动   | 信号    | 阻塞    | 拷贝      |
  | 异步I/O   | 不等待  | 不阻塞  | 无        |
```

## 5. 死锁理论

### 5.1 死锁条件深入

```
死锁四个必要条件 (COFFMAN条件):

1. 互斥 (Mutual Exclusion):
   资源不能被多个进程同时使用
   例: 打印机同一时刻只能被一个进程使用

2. 占有并等待 (Hold and Wait):
   进程持有至少一个资源，同时等待其他资源
   例: 持有资源A，等待资源B

3. 不可抢占 (No Preemption):
   已分配的资源不能被强制剥夺
   例: 不能强制从进程中取走已获得的锁

4. 循环等待 (Circular Wait):
   存在进程等待环: P1等P2, P2等P3, ..., Pn等P1

破坏条件预防死锁:
  破坏1: 允许资源共享 (SPOLLing技术)
  破坏2: 一次性申请所有资源
  破坏3: 允许抢占 (回滚检查点)
  破坏4: 资源有序分配 (编号递增申请)
```

### 5.2 银行家算法

```
银行家算法 (Banker's Algorithm):

  数据结构:
    Available[m]:  每类资源可用数
    Max[n][m]:     每个进程最大需求
    Allocation[n][m]: 每个进程已分配
    Need[n][m] = Max[n][m] - Allocation[n][m]

  安全性算法:
    1. Work = Available, Finish[i] = false (所有i)
    2. 找到 Finish[i]=false 且 Need[i] <= Work 的进程i
    3. 若找到:
         Work = Work + Allocation[i]
         Finish[i] = true
         goto 2
    4. 若所有Finish[i]=true -> 安全; 否则不安全

  示例 (3进程, 3类资源):
    Available = (3, 3, 2)

    进程   Max        Allocation   Need
    P0   (7,5,3)     (0,1,0)     (7,4,3)
    P1   (3,2,2)     (2,0,0)     (1,2,2)
    P2   (9,0,2)     (3,0,2)     (6,0,0)
    P3   (2,2,2)     (2,1,1)     (0,1,1)
    P4   (4,3,3)     (0,0,2)     (4,3,1)

  安全性检查:
    Work = (3,3,2)
    P1: Need(1,2,2) <= Work(3,3,2) -> Work=(5,3,2), Finish[P1]=true
    P3: Need(0,1,1) <= Work(5,3,2) -> Work=(7,4,3), Finish[P3]=true
    P4: Need(4,3,1) <= Work(7,4,3) -> Work=(7,4,5), Finish[P4]=true
    P2: Need(6,0,0) <= Work(7,4,5) -> Work=(10,4,7), Finish[P2]=true
    P0: Need(7,4,3) <= Work(10,4,7) -> Work=(10,5,7), Finish[P0]=true
    安全序列: <P1, P3, P4, P2, P0>

  资源请求算法:
    Request[i] <= Need[i] ?
    Request[i] <= Available ?
    试分配: Available -= Request, Allocation += Request, Need -= Request
    执行安全性检查:
      安全 -> 真正分配
      不安全 -> 回滚试分配，进程等待
```

**银行家算法伪代码**：

```python
def is_safe(available, max_demand, allocation):
    n = len(allocation)
    m = len(available)
    need = [[max_demand[i][j] - allocation[i][j]
             for j in range(m)] for i in range(n)]
    work = available[:]
    finish = [False] * n

    while True:
        found = False
        for i in range(n):
            if not finish[i] and all(need[i][j] <= work[j] for j in range(m)):
                work = [work[j] + allocation[i][j] for j in range(m)]
                finish[i] = True
                found = True
        if not found:
            break
    return all(finish)
```

### 5.3 死锁检测与恢复

```
死锁检测算法:

  与银行家算法类似，但不需要Max矩阵
  只需 Available 和 Allocation

  检测步骤:
    1. Work = Available
    2. 对所有未标记进程, 若 Allocation[i] == 0, 标记
    3. 找到未标记进程i, Request[i] <= Work
    4. Work += Allocation[i], 标记i
    5. 重复3-4直到无法继续
    6. 未标记的进程即为死锁进程

死锁恢复策略:

1. 进程终止:
   - 终止所有死锁进程 (代价大)
   - 逐个终止直到消除死锁 (需选择牺牲者)

2. 资源抢占:
   - 选择牺牲进程: 撤销其持有的资源
   - 回滚: 将牺牲进程回滚到安全状态
   - 饥饿: 同一进程可能反复被选为牺牲者

3. 忽略:
   - UNIX策略: 鸵鸟算法
   - 死锁概率低，恢复代价高
   - 人工重启更经济
```

## 6. Linux内核机制

### 6.1 中断与软中断

```
Linux中断处理:

  硬中断 (Top Half):
    - 立即执行，关中断
    - 只做最紧急的工作(如从网卡复制数据)
    - 不可睡眠

  软中断/Tasklet/工作队列 (Bottom Half):
    - 延后执行，开中断
    - 处理非紧急工作(如协议栈处理)

  中断处理流程:
    硬件中断 -> 保存上下文 -> Top Half处理
                               |
                               v
                          标记软中断
                               |
                               v
                          开中断后执行Bottom Half

  软中断类型 (Linux):
    HI_SOFTIRQ     : 高优先级tasklet
    TIMER_SOFTIRQ  : 定时器
    NET_TX_SOFTIRQ : 网络发送
    NET_RX_SOFTIRQ : 网络接收
    BLOCK_SOFTIRQ  : 块设备
    TASKLET_SOFTIRQ: 普通tasklet
    SCHED_SOFTIRQ  : 调度

  工作队列 (Work Queue):
    在进程上下文中执行，可以睡眠
    适用于需要睡眠的延后工作
```

### 6.2 内核同步机制

```
Linux内核同步原语:

1. 原子操作:
   atomic_t counter = ATOMIC_INIT(0);
   atomic_inc(&counter);
   atomic_dec_and_test(&counter);

2. 自旋锁:
   spinlock_t lock = SPIN_LOCK_UNLOCKED;
   spin_lock(&lock);
   // 临界区 (不可睡眠!)
   spin_unlock(&lock);

   适用: 短临界区, 中断上下文

3. 互斥锁:
   struct mutex mtx;
   mutex_lock(&mtx);
   // 临界区 (可睡眠)
   mutex_unlock(&mtx);

   适用: 长临界区, 进程上下文

4. 读写自旋锁:
   rwlock_t rwlock;
   read_lock(&rwlock);   // 多读者并行
   read_unlock(&rwlock);
   write_lock(&rwlock);  // 写者独占
   write_unlock(&rwlock);

5. RCU (Read-Copy-Update):
   读者无锁访问
   写者创建副本修改，延迟释放旧版本

   // 读者
   rcu_read_lock();
   ptr = rcu_dereference(gptr);
   // 使用ptr
   rcu_read_unlock();

   // 写者
   new_ptr = kmalloc();
   *new_ptr = *old_ptr;  // 复制
   new_ptr->field = new_value;  // 修改副本
   rcu_assign_pointer(gptr, new_ptr);  // 原子替换
   synchronize_rcu();  // 等待所有读者退出
   kfree(old_ptr);  // 释放旧版本

6. 顺序锁 (Seqlock):
   适用于读多写少场景
   读者检测写者干扰后重试
```

### 6.3 进程地址空间管理

```
Linux进程地址空间管理:

  mm_struct (进程内存描述符):
    struct mm_struct {
        pgd_t        *pgd;        // 页全局目录
        unsigned long start_code, end_code;  // 代码段
        unsigned long start_data, end_data;  // 数据段
        unsigned long start_brk, brk;        // 堆
        unsigned long start_stack;           // 栈
        unsigned long mmap_base;             // mmap区域基址
        struct vm_area_struct *mmap;         // VMA链表
        struct rb_root mm_rb;               // VMA红黑树
    };

  VMA (Virtual Memory Area):
    描述一段连续的虚拟内存区域

    struct vm_area_struct {
        unsigned long vm_start;    // 起始虚拟地址
        unsigned long vm_end;      // 结束虚拟地址
        struct file *vm_file;      // 关联文件(mmap)
        unsigned long vm_flags;    // 权限(READ/WRITE/EXEC)
        struct vm_operations_struct *vm_ops;
    };

  缺页处理流程:
    1. CPU触发Page Fault
    2. 找到对应VMA
    3. 检查访问权限
    4. 分配物理页
    5. 若VMA映射文件 -> 从文件读取
    6. 若匿名映射 -> 清零页
    7. 若COW -> 复制页
    8. 建立页表映射
    9. 返回用户态重新执行
```

### 6.4 Linux启动流程

```
Linux启动流程:

  1. BIOS/UEFI
     POST自检 -> 选择启动设备

  2. Bootloader (GRUB2)
     加载内核映像(vmlinuz)和initramfs到内存
     传递启动参数

  3. 内核初始化
     start_kernel()
       -> setup_arch()        // 架构相关初始化
       -> mm_init()           // 内存管理初始化
       -> sched_init()        // 调度器初始化
       -> init_IRQ()          // 中断初始化
       -> timer_init()        // 定时器初始化
       -> rest_init()         // 创建init进程

  4. init进程 (PID=1)
     执行 /sbin/init (或systemd)
     读取 /etc/inittab 或 systemd unit文件
     按运行级别启动服务

  5. 用户空间
     网络服务 -> 登录管理 -> 桌面环境
```
